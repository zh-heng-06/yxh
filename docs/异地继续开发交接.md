## 项目目标

为二手手机门店建设一套苹果、安卓都能使用的局域网库存与经营系统，核心目标是减少手写、重复查询和重复录入。普通固定流程尽量使用本地规则、OCR 和数据库，不调用云端 AI；只有确实需要理解、归纳或辅助判断时才考虑使用 AI，从而控制 token 和费用。

当前采用“先局域网跑通、以后再决定是否云化”的路线。店内 Windows 电脑运行服务，苹果或安卓手机在同一局域网内通过浏览器访问。

## 已经实现的功能

- 首次设置、账号登录、老板与店员权限、修改和重置密码。
- SQLite 本地数据库、WAL 模式、健康检查、备份和恢复。
- 每天19:00校验备份、不可修改的全局操作日志、登录失败限制和磁盘空间检查。
- 爱思助手设备详情截图本地 OCR 入库；截图不上传互联网。
- 苹果和安卓手机手工入库、唯一库存编号、IMEI/序列号校验和原图留档。
- 40×30mm 横向标签、二维码、NIIMBOT B1 本地打印代理、入库后自动打印和补打。
- 库存搜索、详情编辑、库位、预订、送修、退货再售等状态。
- 苹果手机拍照识别二维码，以及库存编号、IMEI、尾号查询后核对出库。
- 出库时记录实际成交价、支付方式、手机壳、手机膜、充电头、充电器和客户备注。
- 每日销售账本，自动计算单台利润：实际成交价减回收成本。
- 销售质保天数、质保到期日、客户售后登记、处理结果和售后成本闭环。
- 超30/60/90天库存、过期预订、久未取机、久未修好和待处理售后提醒。
- 经营报表、CSV 导入导出；盘点界面已按门店当前需求移除。
- 本地规则实现的历史定价参考、销售文案和今日经营总结。
- 第一版行情中心：外部回收/销售行情手工留存、同条件筛选、内外数据分开展示、参考区间与可信度、老板最终定价和调整原因留档。
- 自动化接口与网页测试；现实硬件测试仍应以测试清单为准。

## 市场行情与独立定价

老板目前会参考“手机回收龙猫”“博能二手回收”“数码回收网”等公众号或服务号的回收报价，再对照京东、抖音等平台同型号、容量、成色、电池、拆修和原装情况来决定店内回收价与销售价。

已经确定的产品原则：

1. 外部行情只能作为参考，不能成为唯一价格来源。
2. 门店自己的真实收购、销售、库存天数和利润数据要独立留存，逐步形成自己的定价能力。
3. 不应未经授权抓取公众号或平台私有接口；优先考虑官方接口、人工导入、截图识别和合规的数据采购。
4. 每次建议价都要展示来源、采集时间、样本数量、价格区间和置信度，老板保留最终决定权。
5. 建议未来形成三层价格：外部参考行情、门店历史成交模型、老板策略修正。

2026-07-18 已完成第一版最小可用版本：

- “行情快照”表：按日期保存来源、型号、容量、成色、电池、拆修情况和报价。
- 支持手工录入、删除错误记录，不直接依赖某一个外部平台。
- 行情中心显示外部回收/销售中位数、门店历史成交、当前同款库存和建议回收/销售区间。
- 保存老板最终定价及调整原因，用于以后学习门店自己的定价规律。
- 支持粘贴已确认域名的每日完整报价图片链接，下载原图到本机，以本地 RapidOCR 分段识别“型号、网络型号、容量、靓机、小花、大花、外爆、内爆可测”，人工确认后才批量写入。
- 已接入公开网页 `https://13994040400.huishoubaojia.com/5041.html` 和 `5042.html`：系统每天14:30直接从网页提取当天原图地址，本地识别并自动导入；失败后每小时重试。
- OCR 已改为读取真实横竖表格线、整块识别加缺行放大重试，再按行坐标匹配网络型号。自动导入前强制检查应有行数、型号容量唯一、普通五档齐全和价格递减顺序；5041 另保存可为空的“高保充新”。
- 店内成色与报价表档位已建立第一版对应：全新/99新/95新对应靓机，9成新对应小花，8成新对应大花；原始档位仍单独保存，老板可直接查询。

行情采集代码和历史数据能力继续保留，但首页“行情中心”当前灰色停用。库存营业主流程稳定并积累足够真实数据后，再恢复老板决策台开发；不让 AI 直接自动定价或自动收购。

## 数据与隐私边界

以下内容被 `.gitignore` 排除，不得提交到公开仓库：

- `apps/local-server/data/` 中的真实 SQLite 数据库和备份。
- 顾客手机号、真实 IMEI、序列号、爱思助手截图和手机照片。
- 密码、环境变量、运行日志、打印机本机配置和本地编译文件。

因此，家里电脑从 GitHub 得到的是源代码和文档，不包含店内真实库存，也无法直接使用店里的打印机。家里适合继续设计、编码和自动测试；涉及真实数据、局域网、爱思助手和 NIIMBOT B1 的验收，需要回店里电脑执行。

## 协作约定

- 开始工作前先执行 `git status --short --branch` 和 `git pull --ff-only`。
- 修改前阅读现有代码和测试，不重复搭建另一个原型。
- 不提交真实数据库、账号密码、截图、顾客信息或硬件日志。
- 每项功能完成后先跑自动测试，再更新现实测试清单。
- 家里电脑完成的改动推送 GitHub；店里电脑在确认工作区干净后使用 `git pull --ff-only` 更新。
- 数据库结构修改必须使用可重复执行的迁移或兼容升级逻辑，不能手工改真实数据库。
